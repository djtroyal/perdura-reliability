# Perdura model-assurance framework

## Why this gate exists

Passing tests is not evidence that a scientific model is complete or correct when
the tests restate the implementation.  A model can be internally consistent while
using the wrong likelihood, omitting a termination regime, under-covering its
stated confidence level, or accepting data generated by a different process.

This framework separates six questions that the earlier methodology audit too
often combined:

1. **Identity:** Is the named model actually the method implemented?
2. **Equation correctness:** Does each implemented estimator, statistic, interval,
   and prediction agree with an authoritative derivation?
3. **Capability completeness:** Are every standard-required input form, regime,
   output, diagnostic, and limitation represented, or explicitly listed as a gap?
4. **Statistical calibration:** Do stated confidence levels, tests, and prediction
   intervals achieve their intended operating characteristics in the regimes in
   which Perdura presents them as valid?
5. **Numerical robustness:** Are domain boundaries, tied or sparse observations,
   extreme scales, non-finite input, optimizer failures, and weak identification
   handled without returning plausible-looking invalid results?
6. **End-to-end semantic integrity:** Do the core API, backend schema, UI,
   persistence, plots, exports, and reports preserve the same data-generating
   assumptions, units, parameterization, and uncertainty meaning?

No model may be described as verified merely because its existing tests pass.

## Required assurance record

Each calculation model must have one record in
`docs/audit/model-assurance-matrix.json`.  A record contains:

- implementation entry points and every user-facing route;
- authoritative primary sources with equation, table, or section locators;
- a claim map from every output to its formula and implementation location;
- required and supported input/data regimes;
- independent published benchmarks or analytic identities;
- likelihood-score or estimating-equation checks for fitted models;
- calibration evidence for every inferential claim;
- numerical and adversarial stress evidence;
- API/UI/persistence/report semantic checks;
- a capability inventory and explicit omissions;
- certification status and unresolved blockers.

The matrix also carries a fail-closed `model_inventory_complete` flag.  Domain
coverage alone cannot prove that every model, estimator, option, and alternate
method has a claim record.  That flag is accepted only when a separate,
repository-local `model_inventory_source` enumerates model IDs and reconciles
exactly with the assurance records.  A certified domain may not contain an
uncertified model (or, for a fully verified domain, a caveated model).  The
strict gate cannot pass until the inventory is complete, so a newly added or
previously omitted model cannot disappear behind a passing sample of domain
tests.

Schema v2 uses a typed evidence registry rather than arbitrary path strings.
Each evidence record has a repository-local path, optional test/function node,
evidence kind, independence classification, mapped claim IDs and regimes,
integration layers, and run status. Simulation evidence also identifies its run
profile. Claim applicability entries link back to the same evidence IDs, so an
existing but unrelated file cannot satisfy a claim merely by being named in the
matrix. The checker rejects one-way or inconsistent mappings.

Every claim in a certified model must make an explicit `supported` or
`not_applicable` decision for every required regime. Each supported claim-regime
cell needs passed independent evidence mapped to that exact cell. A PR simulation
profile is a fast regression/calibration smoke test; only a passed `release`
profile can certify an inferential claim. Required end-to-end layers must be
covered collectively by typed integration evidence. Tests that compute their
reference value with the production formula are classified as consistency tests,
not independent validation.

For a certified model, every required capability must also be accounted for
exactly once as implemented or as an explicit gap; a gap cannot be silently
listed as implemented as well.

## Minimum validation matrix

The applicable dimensions must be crossed deliberately rather than sampled only
at a convenient nominal case:

| Dimension | Required examples |
|---|---|
| Sample information | sparse/small, nominal, large |
| Observation design | complete, censored/truncated/grouped, each supported termination rule |
| Parameter regime | boundary/near-boundary, nominal, adverse or weakly identified |
| Scale | very small, ordinary, very large, and unit-rescaled equivalents |
| Data pathology | ties/counts, zeros when allowed, missing/non-finite, clustered observations |
| Model validity | well-specified and representative misspecifications/change points |
| Inference | conditional and unconditional coverage where applicable; failure/completion rate |
| Integration | core, API, UI invalidation, persistence, plot, export, and report |

Dimensions that do not apply require an explicit rationale.  “Not tested” is not
equivalent to “not applicable.”

## Certification states

- `verified`: all required capabilities are implemented, every material claim has
  independent evidence, calibration gates pass, and there are no open blockers.
- `verified_with_caveats`: the implemented scope is correct and its limitations are
  enforced, but named non-critical capabilities remain explicit gaps. Material
  blockers are not allowed in a certified state.
- `needs_revision`: a material correctness, calibration, data-contract, or
  completeness defect exists.
- `not_assessed`: no claim of correctness or completeness is permitted.

The structural checker may pass while models remain unverified; that means the
assurance inventory itself is valid.  A strict release-certification run fails if
any model is not `verified` or `verified_with_caveats`, if a material blocker is
open, or if required evidence is missing.

## Resolution rules

A finding is resolved only when all of the following are true:

1. the source requirement and affected regimes are identified;
2. the implementation is corrected or the unsupported regime is rejected;
3. an independent regression fixture would fail on the old implementation;
4. uncertainty claims have calibration evidence proportional to their risk;
5. numerical failure is explicit and fail-closed;
6. the corrected meaning survives every exposed integration layer;
7. remaining omissions are recorded as gaps, not silently excluded from scope.

Reviewing only the original findings is a regression check, not a new audit.  A
re-audit must also repeat the requirement inventory and search for omitted models,
regimes, and outputs.

## Crow–AMSAA corrective example

The 2026-07 Crow–AMSAA review demonstrates why each gate is required.  Prior tests
confirmed point estimates using the same equations as the implementation, but did
not detect a wrong failure-terminated Cramér–von Mises correction, an invalid
instantaneous-MTBF interval, a non-MLE pooled estimator under unequal exposure,
rejection of a tied handbook benchmark, ambiguous termination semantics, or the
complete absence of the handbook grouped-data method.  Those defects span all six
assurance questions above and cannot be prevented by increasing unit-test count
alone.
